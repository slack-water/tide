# Tide in Practice

*real use cases. not hypotheticals. the system earns credibility through use, not through design.*

*most entries live in `tide-private` — the full context includes personal detail that shouldn't be public. this file contains sanitized versions useful as examples of how tide actually functions.*

---

## use case format

each entry names:
- **the situation** — what was actually happening
- **how tide engaged** — which files were loaded, what the system surfaced
- **the outcome** — what changed, and honestly how much
- **verdict** — what worked, what didn't, what the system couldn't do

---

## 2026-05-23 — 2:30am insomnia spiral

**the situation:** woke at 2:30am mid-insomnia cycle. a single anxiety (finances) triggered a chain of planning: anxiety tracker, ADR framework for tide, blog entry storage system, insomnia documentation pattern. seven action items generated within minutes, none of them addressing the original anxiety.

**how tide engaged:** loaded VALUES.md and FAILURE-MODES.md. the pattern matched immediately: *premature action on discomfort* — moving to planning and action before sitting with what the discomfort actually was. secondary match: *over-optimization spiral* — building structure around the anxiety instead of addressing it. filed six inbox captures and stopped planning.

**the outcome:** partial redirect. didn't go back to sleep — the cost of staying awake wasn't high enough once the anxiety was contained by filing the captures. did do a 40-minute yin yoga class instead of staying up doing system work. that's a real improvement, not a complete one.

**verdict:** useful. the system named the pattern clearly and fast. what it couldn't do is make the right choice feel cheaper than the wrong one — tide lowered the anxiety enough to feel like sufficient action, which removed the pressure to sleep without actually restoring sleep. the gap between *naming a failure mode* and *making it costly enough to override* is real and currently unaddressed. filed as a research item.

## on public use cases

most real tide use cases involve personal context that belongs in `tide-private`. the value of documenting sanitized versions here is to show how the system functions in practice — not as aspirational examples, but as honest records of partial successes, limits, and what the system can and can't do.

the goal isn't to demonstrate that tide works perfectly. it's to accumulate enough real cases to see where it reliably helps, where it falls short, and what would make it better. that feedback loop is what the system is actually for.

if you're forking tide: your use cases belong here. start with your first real use, however messy.
